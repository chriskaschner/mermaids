    - **Tails (3 variants):** tail-1 is the existing tail shape (ocean teal #7ec8c8). Create tail-2 (a wider, flowing tail with scalloped edges) and tail-3 (a split/forked mermaid tail).     - **Hair (3 variants):** hair-1 is the existing flowing hair (lavender #c4a7d7). Create hair-2 (short wavy bob style) and hair-3 (long braided style with a side braid). Each hair group includes head circle and hair paths. All hair groups align at head center (cx=200, cy=80).

    Replace the current hair/tail `<g data-region>` groups with `<use>` elements:
    ```xml
    <use id="active-tail" href="#tail-1" data-region="tail" pointer-events="all" />
    <use id="active-hair" href="#hair-1" data-region="hair" pointer-events="all" />
    ```

    1. active-tail (bottom layer)
    2. body group (middle)

    **Part B: Create dressup.js**

    Create frontend/js/dressup.js as an ES module following the research code example closely:

    Exports:
    - `PARTS` object: `{ tail: ["tail-1", "tail-2", "tail-3"], hair: ["hair-1", "hair-2", "hair-3"], acc: ["acc-none", "acc-1", "acc-2", "acc-3"] }`
    - `COLORS` array: 10 child-friendly hex colors (same as research: ocean teal, lavender, hot pink, gold, sky blue, pale green, tomato red, plum, light salmon, turquoise)
    - `state` object: `{ tail: "tail-1", hair: "hair-1", acc: "acc-none", activeCategory: "tail", colors: {} }`
    - `swapPart(category, variantId)`: Changes `<use>` href, tracks in state, pushes undo, re-applies color override if one exists for the new variant, calls checkCompletion(). No-op if same variant selected.
    - `recolorActivePart(color)`: Gets active category's current variant from state, finds source `<g>` in `<defs>`, updates fill on all fill-bearing child elements (path, circle, ellipse, rect -- skip fill="none"). Stores color override in state.colors[variantId]. Pushes undo that restores previous fills.
    - `undo()`: Pops last command from undoStack and executes its undo function. No-op if stack empty.
    - `checkCompletion()`: Returns true if tail !== "tail-1" AND hair !== "hair-1" AND acc !== "acc-none". Does NOT trigger celebration directly (that's wired in Plan 02's UI layer).
    - `getOriginalColors(variantId)`: Reads fill attributes from source `<g>` children, returns array of {element, fill} for undo support. Called lazily on first recolor of a variant.
    - `resetState()`: Resets state to defaults, clears undo stack. Useful when navigating away and back.

    Undo stack capped at 30 items (MAX_UNDO). When exceeded, shift() the oldest.

    IMPORTANT per research Pitfall 5: Color overrides are stored in state.colors keyed by variantId, NOT mutated globally. When swapPart() switches to a variant that has a color override, it re-applies that override. When switching away, no cleanup needed since the `<use>` element simply points elsewhere.

    Actually -- per research, the color IS applied directly to the source `<g>` elements in `<defs>`. This means Pitfall 5 IS relevant: switching tail-1 to red, then switching to tail-2, then back to tail-1 -- tail-1 should still be red because the `<defs>` source was mutated. BUT switching to tail-2 should show tail-2's ORIGINAL colors, not red. The state.colors map tracks which variants have been recolored so undo can restore correctly. The original fills must be captured before first mutation via getOriginalColors().

    Revised approach for recolorActivePart:
    1. Get variantId from state[activeCategory]
    2. If this variant hasn't been recolored before, capture original fills via getOriginalColors() and store in a `originalColors` map
    3. Apply new color to all fill-bearing elements in the source `<g>`
    4. Store color in state.colors[variantId]
    5. Push undo that restores previous fills (element by element)
