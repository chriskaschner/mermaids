<task type="auto">
  <name>Task 1: Fix z-index layering and add event propagation guards</name>
  <files>frontend/css/style.css, frontend/js/dressup.js</files>
  <action>
**Root cause hypothesis:** On real iPad Safari, the `position: fixed; bottom: 0` nav bar (#nav-bar) and the selection panel (.selection-panel) overlap at the bottom of the viewport. The nav bar has no z-index, and neither does the selection panel. On iOS Safari with `viewport-fit=cover` and safe area insets, the nav bar may sit ON TOP of the selection panel buttons. When the user taps an option button, the tap hits the nav bar `<a href="#/dressup">` anchor instead, causing the dressup view to re-render (which looks like "showing the full mermaid list" / navigating away).

**CSS fix (frontend/css/style.css):**

1. Add `position: relative; z-index: 10;` to the `.dressup-view` rule. This creates a stacking context that puts the entire dressup view (including selection panel) above the nav bar.

2. Add `z-index: 5;` to the `#nav-bar` rule. This ensures the nav bar is explicitly lower than the dressup view's selection panel.

3. Add `position: relative; z-index: 20;` to the `.selection-panel` rule. This ensures the selection panel is definitively above the nav bar even if the dressup-view stacking context is not sufficient.

4. Also add the same `position: relative; z-index: 10;` to `.coloring-view` and `z-index: 20;` to `.coloring-panel` for consistency (prevent the same issue in coloring).

**JS fix (frontend/js/dressup.js):**

In the `renderOptions` function, add `event.stopPropagation()` to both the option button click handlers and the color swatch click handlers. This prevents click events from propagating up through the DOM to any underlying elements (nav links, SVG container, etc.).

For option buttons, change the click handler from:
```javascript
btn.addEventListener("click", () => {
```
to:
```javascript
btn.addEventListener("click", (e) => {
    e.stopPropagation();
```

For color swatches, change:
```javascript
btn.addEventListener("click", () => {
```
to:
```javascript
btn.addEventListener("click", (e) => {
    e.stopPropagation();
```

Also add `e.stopPropagation()` to the category tab click handlers in `initDressUp()`:
```javascript
tab.addEventListener("click", async (e) => {
    e.stopPropagation();
```

And to the undo button handler:
```javascript
undoBtn.addEventListener("click", (e) => {
    e.stopPropagation();
```

These are defensive guards -- they should not affect Playwright tests (which already pass) but will prevent real iPad Safari touch events from propagating to underlying elements.

<task type="auto">
  <name>Task 2: Add removable debug overlay for iPad Safari diagnostics</name>
  <files>frontend/js/app.js</files>
  <action>
Add a debug overlay module to app.js that can be activated by triple-tapping the home nav icon (or by appending `?debug=1` to the URL). The overlay:

1. Creates a fixed-position semi-transparent `<div id="debug-overlay">` at the top of the screen (z-index: 9999, position: fixed, top: 0, left: 0, width: 100%, max-height: 30%, overflow-y: auto, background: rgba(0,0,0,0.8), color: #0f0, font-family: monospace, font-size: 11px, padding: 8px, pointer-events: none for the container).

2. Listens to ALL click, pointerdown, and touchstart events on `document` in capture phase. For each event, logs to the overlay: `[event.type] target=${event.target.tagName}.${event.target.className} id=${event.target.id || '-'} closest_btn=${event.target.closest('button,a')?.tagName || 'none'}`

3. Shows the last 15 events, newest at top.

4. Also logs any `window.onerror` and `unhandledrejection` events.

5. The overlay includes a small "X" close button (with pointer-events: auto) to dismiss it.

**Activation:** Add this function `_initDebug()` to app.js. Call it at the end of the DOMContentLoaded handler if `window.location.search.includes('debug=1')`. Also wire a triple-tap detector on the home nav icon (3 taps within 1 second) that toggles the overlay on/off.

This overlay is a diagnostic tool for the iPad Safari verification. It will be removed in a future cleanup, but for now it provides visibility into what element is receiving taps on the real device.

**Important:** The debug overlay must NOT interfere with normal app operation. All event listeners should be in capture phase with `{ capture: true }` and must NOT call stopPropagation or preventDefault.
