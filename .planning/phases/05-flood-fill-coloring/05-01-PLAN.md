<task type="auto" tdd="true">
  <name>Task 1: Implement scanline flood fill algorithm</name>
  <files>frontend/js/floodfill.js, tests/test_floodfill_unit.py</files>
  <behavior>
    - floodFill(imageData, startX, startY, fillColor, tolerance) fills connected pixels matching the target color at (startX, startY)
    - Fill spreads horizontally and vertically but stops at pixels that differ from target by more than tolerance (per RGBA channel)
    - Fill does not bleed through anti-aliased edges when tolerance is 32 (line art pixels are far from white)
    - Fill handles edge cases: start pixel already matches fill color (no-op), start pixel is on a line (no-op or tiny fill), out of bounds coordinates
    - hexToRgb("#ff69b4") returns [255, 105, 180]
    - hexToRgb("#000000") returns [0, 0, 0]
    - Uses iterative scanline approach (not recursive) to avoid stack overflow on large regions
    - Uses Uint8Array visited bitmap to prevent revisiting pixels
  </behavior>
  <action>
    Create `frontend/js/floodfill.js` as a vanilla ES module exporting `floodFill` and `hexToRgb`.

    The flood fill algorithm:
    1. Get the RGBA of the start pixel as the "target color"
    2. If target color already matches fill color, return early (no-op)
    3. Use a stack-based scanline approach: push start point, then for each point pop from stack, scan left to find scanline start, scan right filling pixels and checking above/below for new scanline seeds
    4. Use a `Uint8Array(width * height)` as visited bitmap
    5. Color comparison: each RGBA channel difference must be <= tolerance
    6. Fill by setting pixel data directly in the ImageData typed array: `data[i] = fr; data[i+1] = fg; data[i+2] = fb; data[i+3] = 255;`

    The `hexToRgb(hex)` helper: parse hex string (e.g. "#ff69b4") to [r, g, b] array using `parseInt(hex.slice(1), 16)` and bit shifting.

    Follow the scanline algorithm pattern from RESEARCH.md Pattern 3 closely. Default tolerance constant: `export const DEFAULT_TOLERANCE = 32;`

    For unit testing: create `tests/test_floodfill_unit.py` that uses Playwright to load a test page, creates a canvas in JS, runs floodFill on synthetic ImageData (e.g., a white canvas with a black border rectangle), and asserts pixel colors changed correctly. This exercises the algorithm through the browser since it's an ES module.
  </action>
  <verify>
    <automated>uv run pytest tests/test_floodfill_unit.py -x</automated>
  </verify>
  <done>floodfill.js exports floodFill and hexToRgb. Unit tests confirm: fill spreads within bounded region, stops at edges with tolerance=32, handles same-color no-op, hexToRgb parses correctly.</done>

  <behavior>
    - initColoringCanvas(svgUrl, container) creates a canvas element (1024x1024, willReadFrequently: true), rasterizes the SVG onto it (white bg + drawImage), and returns { canvas, ctx }
    - handleCanvasTap(x, y) calls floodFill on canvas ImageData at the mapped coordinates with the selected color and current tolerance, then puts the modified ImageData back
    - Before each fill, pushes a getImageData snapshot onto the undo stack (capped at MAX_UNDO=30)
    - undo() pops the last ImageData snapshot and calls putImageData to restore canvas state
    - canUndo() returns true when undo stack has entries, false when empty (for disabled button state)
    - releaseCanvas() sets canvas width/height to 0, nulls ctx reference, removes canvas from DOM, clears undo stack (iPad Safari memory safety)
    - COLORS and COLORING_PAGES arrays preserved exactly as before
    - setSelectedColor/getSelectedColor preserved for app.js compatibility
    - state.selectedColor defaults to COLORS[2] (hot pink)
  </behavior>
  <action>

    New exports to add:
    - `initColoringCanvas(svgUrl, container)` -- async function that:
      1. Creates a canvas element, sets width=1024, height=1024
      2. Gets 2d context with `{ willReadFrequently: true }`
      3. Fills canvas white (`ctx.fillRect`)
      4. Loads SVG as Image, draws onto canvas with `drawImage`
      5. Stores canvas and ctx in module-level variables
      6. Returns `{ canvas, ctx }`

    - `handleCanvasTap(canvasX, canvasY)` -- function that:
      1. Pushes current ImageData onto undo stack (via getImageData)
      2. Gets current ImageData
      3. Calls `floodFill(imageData, canvasX, canvasY, state.selectedColor, FILL_TOLERANCE)`
      4. Puts modified ImageData back with putImageData
      5. Trims undo stack to MAX_UNDO (shift if over)

      1. Sets canvas.width = 0, canvas.height = 0
      2. Removes canvas from DOM if it has a parent
      3. Nulls module-level canvas and ctx references
      4. Clears undo stack

    - `undo()` -- pops ImageData from stack, calls putImageData
