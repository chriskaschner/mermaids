       - `create_region_mask(image_size: tuple[int, int], region_bbox: tuple[int, int, int, int]) -> bytes`:
         Creates an RGBA PNG mask. Opaque (alpha=255) everywhere, transparent (alpha=0) in the region_bbox (x1, y1, x2, y2). Uses Pillow Image.new("RGBA") and pixel manipulation. Returns PNG bytes via BytesIO.
         Note: transparent = area to be replaced, opaque = area to preserve (OpenAI's semantics).

       - Region coordinate definitions for 1024x1024 generation space:
         ```python
         REGIONS = {
             "tail": (200, 500, 824, 1024),   # lower body area
             "hair": (200, 0, 824, 350),       # head/hair area
             "acc":  (300, 0, 724, 250),        # top of head for accessories
         }
         ```
         These are starting estimates. The exact values may need tuning based on generation results, but they proportionally map from the existing mermaid.svg viewBox (400x700) to 1024x1024.

       - `edit_region(base_image_path: Path, mask_bytes: bytes, prompt: str, output_path: Path, *, size=IMAGE_SIZE) -> Path`:
         Check if output_path exists (skip if so). Call client.images.edit() via retry_api_call with model="gpt-image-1", base image file, mask as BytesIO, prompt, size. Decode b64_json, write to output_path. Return output_path.

       - `generate_dressup_variants() -> list[Path]`:
         1. Call generate_base_mermaid() to get/skip base image.
         2. Open base image to get dimensions for mask creation.
         3. For each category in DRESSUP_VARIANTS (tail, hair, acc):
            a. Create mask for that region.
            b. For each variant (e.g., tail-1, tail-2, tail-3):
               Call edit_region() with base image, mask, variant-specific prompt from DRESSUP_VARIANTS.
               Output to GENERATED_PNG_DIR / "dressup" / "parts" / f"{variant_id}.png".
         4. Return list of all generated variant paths.

    1. Create `src/mermaids/pipeline/assemble.py` with:
       - `assemble_mermaid_svg(traced_parts_dir: Path, output_path: Path) -> Path`:
         Uses xml.etree.ElementTree to build a new mermaid.svg with the defs+use structure matching the existing frontend pattern.

         Steps:
         a. Register SVG namespace: `ET.register_namespace("", "http://www.w3.org/2000/svg")`
         b. Create root `<svg>` with viewBox="0 0 400 700", preserveAspectRatio="xMidYMid meet", id="mermaid-svg"
         c. Create `<defs>` element
         d. For each traced SVG in traced_parts_dir (tail-1.svg, tail-2.svg, etc.):
            - Parse the SVG file
            - Create a `<g id="{part_id}">` in defs
            - Copy path/shape elements from the traced SVG into the group
            - Apply a viewBox-scaling transform if needed (traced SVGs may be 1024x1024, target is 400x700). Use `transform="scale(X, Y)"` or compute from source/target viewBox.
         e. Add `<g id="acc-none">` with invisible hit-area rect (matching existing: rect x=170 y=20 width=60 height=40 fill=none stroke=none)
         f. Create the watercolor filter definition (copy from existing mermaid.svg structure, or simplify for kawaii style -- use Claude's discretion on whether watercolor filter suits kawaii style. If not, skip the filter.)
         g. Create the rendering group (with or without filter):
            - `<use id="active-tail" href="#tail-1" data-region="tail" pointer-events="all" />`
            - Body group `<g data-region="body" pointer-events="all">` with torso, arms, shell top paths (these are the non-variant body parts -- extract from existing mermaid.svg or generate a body group from a traced base mermaid)
            - `<use id="active-hair" href="#hair-1" data-region="hair" pointer-events="all" />`
            - `<use id="active-acc" href="#acc-none" data-region="accessory" pointer-events="all" />`
         h. Add face details outside the filter group (eyes, sparkles, smile, cheeks -- same as existing mermaid.svg)
         i. Write to output_path. Return output_path.

         Key design decisions:
         - The body group and face details are hardcoded SVG elements (not from traced parts) since they don't vary. Copy them from the existing mermaid.svg or create kawaii-appropriate versions.
         - Variant groups contain traced path data wrapped in `<g id="...">`.
         - The transform on each variant group may need adjustment based on actual traced SVG coordinates. Start with a scale factor derived from (400/1024, 700/1024) and adjust.

       - `copy_coloring_pages_to_frontend() -> list[Path]`:
         Copies all SVGs from GENERATED_SVG_DIR / "coloring" to FRONTEND_SVG_DIR / "coloring", overwriting existing files. Returns list of destination paths.

       - `copy_mermaid_to_frontend(assembled_path: Path) -> Path`:
         Copies assembled mermaid.svg to FRONTEND_SVG_DIR / "mermaid.svg". Returns destination path.
